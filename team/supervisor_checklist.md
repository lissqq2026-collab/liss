# 老督审查清单

## 触发方式

用户可以随时说以下任意一句话触发监管审查：
- "老督，审查一下"
- "帮我看看团队的工作质量"
- "出一份审查报告"
- "检查一下最近的代码"

Team Lead 在每个重要里程碑完成后，也应主动触发老督审查。

---

## 代码质量审查项

### 必查（零容忍）
- [ ] 是否存在硬编码的密钥/密码/Token
- [ ] 是否存在 SQL 拼接（注入风险）
- [ ] 是否存在未处理的用户输入直接渲染（XSS风险）
- [ ] 核心功能是否完全没有测试

### 常规检查
- [ ] 变量/函数命名是否语义清晰
- [ ] 是否有无意义的重复代码（DRY原则）
- [ ] 错误处理是否完整
- [ ] 注释是否只出现在必要位置
- [ ] 函数/模块职责是否单一

---

## 工作状态审查项

- [ ] 任务是否按分工完成，没有越界或甩锅
- [ ] 输出内容是否完整（不缺角、不草率）
- [ ] 是否有模糊敷衍的表述（"大概"、"应该可以"、"差不多"）
- [ ] QA 发现的 Bug 是否有明确修复记录
- [ ] Team Lead 的分工是否合理均衡

---

## 评分标准

| 分数 | 含义 |
|------|------|
| 5 | 超出预期，主动发现问题或提供额外价值 |
| 4 | 完成质量高，有少量可改进点 |
| 3 | 合格完成，存在明显但非严重的问题 |
| 2 | 勉强完成，有较多问题需要改进 |
| 1 | 严重不足，输出不可用或存在安全风险 |
